Quick and general overview
Based on simple REST principles, our HTTP 1.1 API endpoints interacts with the
metadata about phones (a picture, brand name, model name, os, and screen size)
from our SQLite database.
Our API accepts the following request methods:
-
GET
-
POST
-
PUT
-
DELETE
This means that our GET, PUT and DELETE requests are idempotent, which means
that the intended effect on the server of multiple identical requests with that method
is the same as the effect for a single such request. The GET method is also safe.
The base address of the RESTful HTTP API is https://localhost:3000 . The API
provides via the /api path a set of endpoints, each with its own unique path, which
allows the user to interact with our database. Our API doesn’t require the user to use
an API key or undergo some sort of authorization process. Cross-Origin requests are
enabled.

Our API is based on REST principles and supports all CRUD operations. Data
resources are accessed via standard HTTP requests in UTF-8 format to an API
endpoint. Data in the response body is in the application/json MIME format. Where
possible, the API uses appropriate HTTP verbs for each action:
STATUS CODE
DESCRIPTION

read the result of the request in the body and the
headers of the response.

processing, but the processing has not been
completed.

no message body.
400
Bad Request / Server Error - The message body will
contain more information.
404
Not Found - The requested resource could not be
found. This error can be due to a temporary or
permanent condition.

GET with specific ID
Endpoint
Request method
Description
/api/phones/id/:id
GET
Get only phone with ID “id”
The user will have to enter the ID of the phone he/she is looking for in the URL of the
request. As a response to this request, the webserver will return a JSON file
containing the metadata of the phone with the corresponding ID in the database In
case of success, the response will have a status code of 200 and the body will look
like the following example:
{
"id": 1,
"brand": "Apple",
"model": "iPhone 11",
"os": "IOS",
"image":
"https://fra1.digitaloceanspaces.com/cdn-armor3/armor3/product_images/2019/12/20/
iphone_11_paars.png",
"screensize": 6
}
In case the user enters an ID while there is no corresponding ID in the database, the
server will respond with a header with status code 404 and the body will contain the
following error message in JSON:
{
"Error": "Entry not found."
}
In case of any other (internal) errors, the server will respond with a header with
status code 400 and the body will contain the error message in JSON.
POST
Endpoint
Request method
Description
/api/phones
POST
Create a new phone row in database
This request is used to add data to our database. This data would be the key/value
pairs of the brand, model, os, image and screensize. The data should be present in
the request body in JSON. Example of JSON POST request body:
{
"brand": "Apple",
"model": "iPhone 11",
"os": "IOS",
"image":
"https://fra1.digitaloceanspaces.com/cdn-armor3/armor3/product_images/2
019/12/20/iphone_11_paars.png",
"screensize": 6
}
In case of success, the web server will return a 201 status code and JSON data in
the body containing the automatically generated id of the corresponding phone which
is now added to the database. This id can later be used to remove/get/update this
specific row’s data. Example:
{
"id": "6"
}
In case one or more of the key/value pairs have an empty string as value or the
screen size is not an integer, the server will respond with a 400 error code with either
one of these JSON messages in the response body:
{
{"Error":"One or more of the passed string values are empty"}
}
{
{"Error":"Enter a valid screensize"}
}
In case of any other (internal) errors, the server will respond with a header with
status code 400 and the body will contain the error message in JSON.

The user will have to enter the ID of the phone he/she is looking for in the URL of the

"brand": "Apple",
"model": "iPhone 11",
"os": "IOS",

}
In case of success, the web server will return a 204 status code with no message.
In case the user enters an ID while there is no corresponding ID in the database, the
server will respond with a header with status code 404 and the body will contain the
following error message in JSON:
{
"Error": "Entry not found."
}
In case one or more of the key/value pairs have an empty string as value or the
screen size is not an integer, the server will respond with a 400 error code with either
one of these JSON messages in the response body:
{
{"Error":"One or more of the passed string values are empty"}
}
{
{"Error":"Enter a valid screensize"}
}
In case of any other (internal) errors, the server will respond with a header with
status code 400 and the body will contain the error message in JSON.
